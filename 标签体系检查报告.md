## 一、标签体系架构总览

### 1.1 标签体系演进

项目经历了从**原五级标签体系**到**新三分法标签体系**的演进：

#### 原标签体系（历史）
```
tag_level_1 (一级标签 - 身份)     → 用户身份标识
tag_level_2 (二级标签 - 话题分类) → 话题内容分类
tag_level_3 (三级标签 - 地点)     → 位置信息
tag_level_4 (四级标签 - 自定义)   → 用户自定义话题标签
tag_level_5 (五级标签 - 商品)     → 商品分类
```

#### 新标签体系（当前推荐）
```
topic_tag      → 话题标签（包含身份 + 话题分类）
product_tag    → 商品标签（交易中心使用）
location_tag   → 地点标签（位置标记）
tag_level_4    → 保留用于用户自定义话题标签
```

## 二、数据库表结构分析

### 2.1 核心标签表

#### 1. `tag_level_1` - 身份标签表（保持不变）
**用途**: 存储用户身份类型  
**字段**:
- `code`: 身份代码 (student, merchant, organization, admin, individual)
- `name`: 身份名称
- `icon`: 图标 emoji
- `sort_order`: 排序
- `is_active`: 是否启用

**数据示例**:
```sql
student    - 学生 👨‍🎓
merchant   - 商户 💼
organization - 团体 👥
admin      - 管理员 ⚙️
individual - 社会 🌍
```

#### 2. `tag_level_2` - 话题分类标签表（保持不变）
**用途**: 存储话题的官方分类  
**字段**:
- `code`: 分类代码 (study_experience, tech_exchange, campus_life, etc.)
- `name`: 分类名称
- `color`: 标签颜色
- `icon`: 图标
- `sort_order`: 排序
- `is_active`: 是否启用

**数据示例**:
```sql
study_experience - 学习经验 #50C878
tech_exchange    - 技术交流 #4169E1
campus_life      - 校园生活 #FF6B6B
food_recommend   - 美食推荐 #FFA07A
activity         - 社团活动 #DDA0DD
employment       - 就业信息 #98FB98
help             - 求助帖 #F0E68C
confession       - 表白墙 #FFB6C1
```

#### 3. `tag_level_3` - 地点标签表（保持不变）
**用途**: 存储校园地点信息  
**字段**:
- `code`: 地点代码 (library, cafeteria, dormitory, etc.)
- `name`: 地点名称
- `location_type`: 地点类型 (campus/building/facility/area/other)
- `latitude/longitude`: 经纬度
- `address`: 详细地址
- `icon`: 图标
- `sort_order`: 排序
- `is_active`: 是否启用

**数据示例**:
```sql
library       - 图书馆 📚
cafeteria     - 食堂 🍽️
dormitory     - 宿舍 🏠
playground    - 操场 🏟️
teaching_building - 教学楼 🏫
laboratory    - 实验室 🔬
```

#### 4. `tag_level_4` - 用户自定义话题标签表（保留）
**用途**: 用户创建的话题标签，需审核  
**字段**:
- `name`: 标签名称
- `category`: 所属分类 (tech/study/life/hobby)
- `usage_count`: 使用次数
- `trend_score`: 趋势分数
- `created_by`: 创建者用户 ID
- `status`: 状态 (active/banned/pending)
- `sort_order`: 排序
- `is_active`: 是否启用

**特点**: 
- 用户发布话题时可创建自定义标签
- 需要管理员审核后才能公开使用
- 支持搜索和推荐

#### 5. `tag_level_5` - 商业标签表（新增）
**用途**: 商品相关的商业标签  
**字段**:
- `code`: 标签代码
- `name`: 标签名称
- `category`: 商品分类
- `icon`: 图标
- `color`: 颜色
- `usage_count`: 使用次数
- `trend_score`: 趋势分数
- `status`: 状态 (active/banned/pending)
- `sort_order`: 排序
- `is_active`: 是否启用

**数据示例**:
```sql
secondhand  - 二手物品
books       - 教材书籍
digital     - 数码产品
clothing    - 服装鞋帽
daily       - 生活用品
sports      - 体育用品
service     - 服务需求
parttime    - 兼职信息
food        - 美食外卖
```

#### 6. `topic_tag` - 话题标签表（新体系核心）
**用途**: 统一存储话题相关的所有标签（身份 + 话题分类）  
**字段**:
- `name`: 标签代码/名称
- `type`: 标签类型 (system/custom)
- `usage_count`: 使用次数
- `trend_score`: 趋势分数
- `last_used_at`: 最后使用时间
- `created_by`: 创建者用户 ID（用户自定义）
- `status`: 状态 (active/banned/pending)
- `icon`: 图标 URL
- `color`: 标签颜色
- `sort_order`: 排序
- `is_active`: 是否启用

**数据来源**:
- 从 `tag_level_1` 复制身份标签
- 从 `tag_level_2` 复制话题分类标签

#### 7. `product_tag` - 商品标签表（新体系）
**用途**: 商品分类标签  
**字段**: 与 `topic_tag` 相同

**数据来源**:
- 从 `tag_level_5` 复制商业标签

#### 8. `location_tag` - 地点标签表（新体系）
**用途**: 地点标记标签  
**字段**: 与 `topic_tag` 相同

**数据来源**:
- 从 `tag_level_3` 复制地点标签

## 五、API 接口分析

### 5.1 公共标签 API

**基础路径**: `/api/tags`

#### 话题标签相关
```
GET /api/tags/topics          → 获取所有话题标签
GET /api/tags/level1          → 获取身份标签（一级标签）
GET /api/tags/topics/hot      → 获取热门话题标签
POST /api/tags/topics/custom  → 创建自定义话题标签
```

#### 商品标签相关
```
GET /api/tags/products        → 获取所有商品标签
GET /api/tags/products/hot    → 获取热门商品标签
POST /api/tags/products/custom → 创建自定义商品标签
```

#### 地点标签相关
```
GET /api/tags/locations       → 获取所有地点标签
GET /api/tags/locations/hot   → 获取热门地点标签
POST /api/tags/locations/custom → 创建自定义地点标签
```

#### 管理员功能
```
POST /api/tags/{type}/{id}/audit    → 审核标签
POST /api/tags/{type}/{id}/convert  → 转为系统标签
```

### 5.2 管理员标签 API

**基础路径**: `/api/admin/tags`

```
GET  /admin/tags/level-2      → 分页查询二级标签

PUT  /admin/tags/level-2/{id} → 更新二级标签
DELETE /admin/tags/level-2/{id} → 删除二级标签
```

## 六、标签应用场景

### 6.1 发布话题场景

**流程**:
1. 用户进入发布页面
2. **身份标签**: 根据用户身份自动选择（只读）
3. **话题分类**: 从二级标签选择最多 3 个
4. **地点标签**: 可选，从三级标签选择最多 2 个
5. **自定义标签**: 可选，从四级标签池选择或创建最多 5 个
6. 提交后，后端增加标签使用次数

**数据结构**:
```json
{
  "level1Tag": "student",
  "level2Tags": ["study_experience", "tech_exchange"],
  "level3Tags": ["library"],
  "level4Tags": [{"name": "Java", "category": "tech"}]
}
```

### 6.2 发布商品场景

**流程**:
1. 用户进入商品发布页面
2. 点击"选择/自定义标签"按钮
3. 从商品标签池选择或创建标签
4. 已选标签以紫色渐变 chips 展示
5. 提交后，后端增加标签使用次数

**数据结构**:
```json
{
  "productTags": ["secondhand", "digital"]
}
```

### 6.3 话题筛选场景

**话题墙筛选**:
1. 点击一级标签（身份）筛选发布者身份
2. 点击二级标签（话题分类）筛选内容类型
3. 点击三级标签（地点）筛选位置
4. 支持多选和清除筛选

### 6.4 商品筛选场景

**交易中心筛选**:
1. 通过下拉框选择身份标签
2. 通过标签 chips 选择商品分类
3. 支持清空筛选

## 七、数据迁移策略

### 7.1 从旧标签表到新标签表

**SQL 迁移脚本**: `check_and_fix_tags.sql`

#### 迁移步骤:

1. **检查现有数据**
```sql
-- 检查原标签表
SELECT COUNT(*) FROM tag_level_1;
SELECT COUNT(*) FROM tag_level_2;
SELECT COUNT(*) FROM tag_level_3;
SELECT COUNT(*) FROM tag_level_5;

-- 检查新标签表

```

2. **执行数据迁移**
```sql
-- 从 tag_level_1 复制到 topic_tag（身份标签）
INSERT INTO topic_tag (name, type, status, is_active, sort_order, icon, color)
SELECT code, 'system', 'active', is_active, sort_order, icon, color
FROM tag_level_1;

-- 从 tag_level_2 复制到 topic_tag（话题分类标签）
INSERT INTO topic_tag (name, type, status, is_active, sort_order, icon, color)
SELECT code, 'system', 'active', is_active, sort_order, icon, color
FROM tag_level_2;

-- 从 tag_level_3 复制到 location_tag（地点标签）
INSERT INTO location_tag (name, type, status, is_active, sort_order, icon, color)
SELECT code, 'system', 'active', is_active, sort_order, icon, NULL
FROM tag_level_3;

-- 从 tag_level_5 复制到 product_tag（商品标签）
INSERT INTO product_tag (name, type, status, is_active, sort_order, icon, color)
SELECT code, 'system', 'active', is_active, sort_order, icon, color
FROM tag_level_5;
```

3. **验证迁移结果**

## 九、标签审核机制

### 9.1 用户自定义标签审核流程

1. **创建阶段**
   - 用户创建自定义标签
   - 初始状态：`pending`（待审核）
   - 仅创建者可见

2. **审核阶段**
   - 管理员在后台查看待审核标签
   - 可以批准（`active`）或拒绝（`banned`）

3. **使用后**
   - 审核通过的标签进入标签池
   - 其他用户可以选择使用
   - 系统记录使用次数和趋势分数

### 9.2 热门标签算法

**趋势分数计算**:
```java
tag.setTrendScore(tag.getUsageCount() * 0.1);
```

**热门标签获取**:
```java
QueryWrapper<TopicTag> wrapper = new QueryWrapper<>();
wrapper.eq("is_active", true)
       .and(w -> w.eq("type", "system").or().eq("status", "active"));
wrapper.orderByDesc("trend_score");
wrapper.last("LIMIT 10");
```

## 十、存在的问题和建议

### 10.1 发现的问题

1. **标签体系双轨制**
   - 同时存在原五级标签和新三分法标签
   - 可能导致数据冗余和维护困难

2. **前端硬编码问题**
   - Topicwall.vue 中 displayLevel1Tags 等使用硬编码
   - 应该优先使用后端返回的数据

3. **缺少标签关联关系**
   - 话题与标签是多对多关系
   - 但 topics 表使用 JSON 数组存储标签代码
   - 不利于复杂查询和统计

4. **标签使用统计不完善**
   - 有 usage_count 和 trend_score 字段
   - 但没有详细的统计分析功能

### 10.2 改进建议

1. **统一标签体系**
   - 逐步淘汰 tag_level_1~5 的命名方式
   - 统一使用 topic_tag, product_tag, location_tag

2. **建立标签关联表**
   ```sql
   CREATE TABLE topic_tag_relation (
     topic_id BIGINT,
     tag_id INT,
     tag_type VARCHAR(20),  -- identity/topic/location/custom
     PRIMARY KEY (topic_id, tag_id)
   );
   ```

3. **增强标签统计**
   - 记录每日使用量
   - 分析热门标签趋势
   - 支持标签推荐算法

4. **优化用户体验**
   - 添加标签搜索功能
   - 支持标签同义词
   - 提供标签使用指南

## 十一、总结

### 11.1 标签体系现状

项目当前采用**混合标签体系**：
- **保留**: tag_level_1~5 作为数据源和管理入口
- **新建**: topic_tag, product_tag, location_tag 作为统一查询接口
- **兼容**: 前端组件同时支持两种体系

### 11.2 核心设计思想

1. **分类管理**: 按业务场景分为话题、商品、地点三大类
2. **系统 + 自定义**: 支持官方标签和用户自定义标签
3. **审核机制**: 用户自定义标签需审核，确保质量
4. **热度算法**: 基于使用次数和趋势分数推荐热门标签

### 11.3 技术亮点

- ✅ 多层级标签选择器，支持复杂业务需求
- ✅ 标签颜色规范化，提升视觉识别度
- ✅ 审核机制完善，保证标签质量
- ✅ 热门标签算法，促进优质内容传播
- ✅ 前后端分离，API 设计规范
