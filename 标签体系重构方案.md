## 二、重构方案

### 方案一：快速修复（推荐）✅

**核心思路：** 将 TagServiceImpl 中的查询转发到新的 Service

#### 2.1.1 需要修改的文件

1. **TagServiceImpl.java**
   - 将 `getLevel1Tags()` 转发到 `IdentityTagsService`
   - 将 `getLevel2Tags()` 转发到 `TopicTagService`
   - 将 `getLevel3Tags()` 转发到 `LocationTagService`
   - 保留 `getLevel4Tags()` 直接操作 tag_level_4 表
   - 删除所有 `getLevel5Tags()` 相关方法

2. **AdminTagController.java**
   - 标记为 `@Deprecated`
   - 建议迁移到新的管理接口

3. **删除废弃文件**
   - 删除实体类：`TagLevel1.java`, `TagLevel2.java`, `TagLevel3.java`, `TagLevel5.java`
   - 删除 Mapper：`TagLevel1Mapper.java`, `TagLevel2Mapper.java`, `TagLevel3Mapper.java`, `TagLevel5Mapper.java`

#### 2.1.2 优点
- ✅ 改动最小
- ✅ 风险最低
- ✅ 保持向后兼容
- ✅ 前端无需修改

#### 2.1.3 缺点
- ⚠️ TagServiceImpl 仍然保留，代码冗余
- ⚠️ AdminTagController 需要后续重构

### 方案二：彻底重构

**核心思路：** 完全删除 TagService，直接使用新的 Service

#### 2.2.1 需要修改的文件

1. **删除文件**
   - `TagService.java`
   - `TagServiceImpl.java`
   - 所有 TagLevelX 实体类和 Mapper

2. **修改 AdminTagController.java**
   - 注入新的 Service
   - 重写所有方法使用新表

3. **修改前端**
   - 更新 API 调用路径
   - 更新数据结构处理

#### 2.2.2 优点
- ✅ 代码清晰，无历史包袱
- ✅ 架构统一，易于维护

#### 2.2.3 缺点
- ❌ 改动巨大，风险高
- ❌ 需要全面测试
- ❌ 工作量大

```java
// 已完成的修改
@Override
public ApiResult getLevel1Tags() {
    // 从 identity_tags 表获取身份标签
    QueryWrapper<IdentityTags> wrapper = new QueryWrapper<>();
    wrapper.eq("is_active", true).orderByAsc("sort_order");
    List<IdentityTags> tags = identityTagsMapper.selectList(wrapper);
    return ApiResult.success(tags);
}

@Override
public ApiResult getLevel2Tags() {
    // 从 topic_tag 表获取话题分类标签（type=system）
    QueryWrapper<TopicTag> wrapper = new QueryWrapper<>();
    wrapper.eq("type", "system").eq("is_active", true).orderByAsc("sort_order");
    List<TopicTag> tags = topicTagMapper.selectList(wrapper);
    return ApiResult.success(tags);
}

@Override
public ApiResult getLevel3Tags() {
    // 从 location_tag 表获取地点标签
    QueryWrapper<LocationTag> wrapper = new QueryWrapper<>();
    wrapper.eq("type", "system").eq("is_active", true).orderByAsc("sort_order");
    List<LocationTag> tags = locationTagMapper.selectList(wrapper);
    return ApiResult.success(tags);
}
```

## 四、数据库表结构对比

### 4.1 原 tag_level_X 表结构

```sql
-- tag_level_1 (身份)
CREATE TABLE tag_level_1 (
  id INT AUTO_INCREMENT PRIMARY KEY,
  code VARCHAR(50) NOT NULL UNIQUE,
  name VARCHAR(100) NOT NULL,
  icon VARCHAR(100),
  sort_order INT DEFAULT 0,
  is_active TINYINT(1) DEFAULT 1
);

-- tag_level_2 (话题分类)
CREATE TABLE tag_level_2 (
  id INT AUTO_INCREMENT PRIMARY KEY,
  code VARCHAR(50) NOT NULL UNIQUE,
  name VARCHAR(100) NOT NULL,
  icon VARCHAR(100),
  color VARCHAR(20),
  sort_order INT DEFAULT 0,
  is_active TINYINT(1) DEFAULT 1
);

-- tag_level_3 (地点)
CREATE TABLE tag_level_3 (
  id INT AUTO_INCREMENT PRIMARY KEY,
  code VARCHAR(50) NOT NULL UNIQUE,
  name VARCHAR(100) NOT NULL,
  location_type VARCHAR(20),
  latitude DECIMAL(10, 8),
  longitude DECIMAL(11, 8),
  icon VARCHAR(100),
  color VARCHAR(20),
  sort_order INT DEFAULT 0,
  is_active TINYINT(1) DEFAULT 1
);
```

### 4.2 新标签表结构

```sql
-- identity_tags (身份标签)
CREATE TABLE identity_tags (
  id INT AUTO_INCREMENT PRIMARY KEY,
  code VARCHAR(50) NOT NULL UNIQUE,
  name VARCHAR(100) NOT NULL,
  icon VARCHAR(100),
  sort_order INT DEFAULT 0,
  is_active TINYINT(1) DEFAULT 1
);

-- topic_tag (话题标签)
CREATE TABLE topic_tag (
  id INT AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  type VARCHAR(20) NOT NULL DEFAULT 'custom',
  usage_count BIGINT DEFAULT 0,
  trend_score DOUBLE DEFAULT 0.0,
  status VARCHAR(20) DEFAULT 'pending',
  icon VARCHAR(100),
  color VARCHAR(20),
  is_active TINYINT(1) DEFAULT 1,
  sort_order INT DEFAULT 0
);

-- location_tag (地点标签)
CREATE TABLE location_tag (
  id INT AUTO_INCREMENT PRIMARY KEY,
  code VARCHAR(50) NOT NULL UNIQUE,
  name VARCHAR(100) NOT NULL,
  type VARCHAR(20) NOT NULL DEFAULT 'system',
  icon VARCHAR(100),
  color VARCHAR(20),
  is_active TINYINT(1) DEFAULT 1,
  sort_order INT DEFAULT 0
);

-- product_tag (商品标签)
CREATE TABLE product_tag (
  id INT AUTO_INCREMENT PRIMARY KEY,
  code VARCHAR(50) NOT NULL UNIQUE,
  name VARCHAR(100) NOT NULL,
  category VARCHAR(50),
  icon VARCHAR(100),
  color VARCHAR(20),
  usage_count BIGINT DEFAULT 0,
  trend_score DOUBLE DEFAULT 0.0,
  status VARCHAR(20) DEFAULT 'active',
  sort_order INT DEFAULT 0,
  is_active TINYINT(1) DEFAULT 1
);
```

## 五、迁移映射关系

| 原表 | 新表 | 数据迁移 | 备注 |
|------|------|----------|------|
| tag_level_1 | identity_tags | 直接复制 | 字段完全一致 |
| tag_level_2 | topic_tag (type=system) | 部分字段 | 新增 type、usage_count 等字段 |
| tag_level_3 | location_tag (type=system) | 部分字段 | 简化了地点信息 |
| tag_level_5 | product_tag | 直接复制 | 字段基本一致 |
| tag_level_4 | topic_tag (type=custom) | 概念转换 | 用户自定义标签转为 topic_tag |
